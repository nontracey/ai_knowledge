# 4.11 AutoGen 多智能体框架

## AutoGen 概述

### 什么是 AutoGen

AutoGen 是微软推出的开源框架，专为构建基于大语言模型的复杂应用而设计。核心思想是通过智能体间的对话与协作来动态驱动任务解决，将任务分解为多个子任务，分配给不同智能体，实现高效协作。架构优势体现在任务分解和角色专业化，提升系统灵活性和可扩展性。

### 与 LangChain 和 SK 对比

| 维度 | LangChain | Semantic Kernel | AutoGen |
|------|-----------|-----------------|---------|
| 核心抽象 | 链/图 | 内核与插件 | 对话 |
| 多智能体 | 通过 Agent 实现 | 通过 Agent 实现 | 原生支持 |
| 交互模式 | 链式调用 | 函数调用 | 智能体对话 |
| 学习曲线 | 中等 | 中等 | 较高 |

AutoGen 的设计理念以"多智能体对话"为中心，对话不仅是信息交换的形式，更是整个应用框架的计算核心。通过对话结构，开发者可以管理复杂的控制流，包括条件逻辑、循环和错误处理。

## AutoGen 分层架构设计

### 分层架构概览

| 层次 | 说明 | 目标用户 |
|------|------|---------|
| Studio | 无代码 UI，用于快速原型设计和测试多智能体工作流 | 产品经理、AI 应用设计师 |
| AgentChat | 高级 API，用于快速开发任务驱动的多智能体对话应用 | AI 应用开发者 |
| Core | 底层 API，事件驱动的框架，用于构建自定义、可扩展的智能体系统 | 系统架构师、框架开发者 |
| Extensions | 插件库，提供模型客户端、工具等可插拔功能 | 所有开发者 |

### AgentChat 与 Core 的关系

AgentChat 通过其"有偏见"的设计为开发者提供便利性，大部分编排逻辑已被封装在预置组件中，适合快速开发常见应用。

Core 作为"无偏见"的底层 API，将编排逻辑的控制权完全交给开发者，允许实现任何交互模式或集成到外部事件驱动架构中。

**核心差异解读：**

| 维度 | AgentChat | Core |
|------|-----------|------|
| 设计理念 | 有偏见（预置编排逻辑） | 无偏见（完全开放控制） |
| 核心优势 | 开发便利、开箱即用 | 极致灵活、完全可控 |
| 适用场景 | 快速开发常见多智能体对话应用 | 自定义交互模式、集成外部事件驱动系统 |
| 开发者定位 | 大多数 AI 应用开发者 | 系统架构师、框架开发者、专家级用户 |

## 核心概念

### 可对话智能体

可对话智能体是一个能够发送和接收消息并基于消息产生回应的实体，是**状态化**和**消息驱动**的：

- **状态化**：智能体能够在多次交互中维持内部状态（如对话历史或任务理解），处理长周期的连贯任务
- **消息驱动**：智能体的所有行为都是由接收到的消息触发的，可以生成自然语言回复、调用外部工具或请求人类用户反馈

### 智能体内部机制

智能体接收到消息后，会处理传入消息列表，结合内部对话历史理解上下文，基于预设角色、工具集和当前状态决定下一步行动，并生成出站消息作为回应。

### 预设智能体角色

- **助手智能体（AssistantAgent）**：由大语言模型驱动，其行为由系统提示和工具集定义，是执行任务的核心力量
- **用户代理智能体（UserProxyAgent）**：是人类用户的代理人，负责发起对话并将用户请求传递给其他智能体，同时作为人机交互的桥梁，能够代表用户执行代码并反馈结果

这两种智能体在一问一答中构成 AutoGen 中最经典的双智能体协作模式。

## 对话模式

### 双智能体对话模式

双智能体对话模式模拟两个人之间的一对一交流，由一个智能体发起对话，将初始任务描述发送给另一个智能体，接收方根据配置生成回复并发送回发起方，形成轮流发言的循环。对话循环会在满足预设终止条件时结束（如达到最大轮次或包含特定终止关键词）。

```python
import asyncio
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.ui import Console
from autogen_ext.models.openai import OpenAIChatCompletion

model_client = OpenAIChatCompletion(model="gpt-4")

assistant = AssistantAgent(
    name="assistant",
    model_client=model_client,
    tools=[weather_tool]
)

user_proxy = UserProxyAgent(name="user")

async def main():
    result = await assistant.run_chat(
        task="北京今天的天气怎么样？适合出门吗？",
        user_proxy=user_proxy
    )
    print(result)

asyncio.run(main())
```

### 顺序对话模式

顺序对话是一连串双智能体对话的链式组合，前一个对话的输出成为后一个对话的输入，通过"接续信息"实现任务上下文的传递。这种模式适合步骤固定、依赖关系明确的复杂任务，如内容创作流程，能够确保最终产出物的连贯性和高质量。

```python
# 链式对话
result1 = await agent1.run_chat(task="任务1")
result2 = await agent2.run_chat(task=f"基于{result1}执行任务2")
result3 = await agent3.run_chat(task=f"基于{result2}执行任务3")
```

### 群聊对话模式

群聊允许多个智能体在一个共享对话空间中交流，发言顺序动态决定，引入群聊管理器作为核心角色。群聊管理器集中控制对话流程，决定每次消息广播后由谁发言，通过智能选择发言者引导对话朝着解决问题的方向发展。

```python
from autogen_agentchat.teams import GroupChat

team = GroupChat(
    participants=[agent1, agent2, agent3],
    max_turns=10
)

await team.run(task="完成一个复杂项目")
```

## 团队协作

### AgentChat 团队的定义与优势

AgentChat 团队是一组智能体集合，它们为了达成共同目标而协同工作。这种团队模式通过多智能体协作，能够有效克服单一智能体的性能瓶颈，实现任务的高效分解和模块化。

### 单一智能体的困境

单一智能体在面对复杂任务时存在明显局限性：在规划、执行、观察和反思等多个阶段难以兼顾，导致系统提示臃肿和混乱。将复杂任务中的不同职责分离给多个智能体是解决这一困境的关键。

### 轮询式小组对话

轮询式小组对话的工作原理是所有成员共享同一个对话上下文，并按顺序轮流发言。这种模式的核心是"对话即状态"，即共享的对话历史记录本身就是团队的共享状态。优点是完全可预测，缺点是缺乏灵活性。

```python
from autogen_agentchat.teams import RoundRobinGroupChat

team = RoundRobinGroupChat(
    participants=[writer, reviewer]
)
```

### 选择式小组对话

选择式小组对话通过大语言模型的"选择器"动态决定下一个发言的智能体。选择器的决策依据包括对话历史、候选智能体的名称和描述，能够根据上下文灵活选择合适的智能体。

```python
from autogen_agentchat.teams import SelectorGroupChat

team = SelectorGroupChat(
    participants=[writer, reviewer, editor],
    selector_func=selector_function  # 自定义选择逻辑
)
```

**轮询式 vs 选择式对比：**

| 特性 | 轮询式小组对话 | 选择式小组对话 |
|------|--------------|--------------|
| 发言者选择机制 | 基于预设的列表顺序，固定轮流发言 | 基于大语言模型对上下文的理解，动态选择发言者 |
| 对话流程 | 线性、可预测、僵化，流程固定 | 动态、自适应、灵活 |
| 适用场景 | 流程固定的任务，如"生成-审查"循环 | 流程不确定的复杂任务 |
| 实现复杂度 | 极低，仅需定义智能体顺序 | 较高，需精心设计智能体描述和选择器提示 |
| 控制方式 | 通过智能体在列表中的位置进行硬编码控制 | 通过自然语言描述和提示进行软编码控制 |

## 用户代理智能体实现人机协作

### 人机协作的必要性

在某些任务中，人类的监督、决策和领域知识至关重要。人机协作能够充分发挥人类和智能体的优势，提升任务的准确性和可靠性。主要应用场景包括：监督与纠偏、提供外部信息、处理模糊性和最终审批。

### 用户代理智能体的实现

用户代理智能体作为人类用户在团队中的"代理人"，在需要时暂停团队流程并等待人类输入。这种机制确保了人类可以随时介入，提供关键决策。

```python
from autogen_agentchat.agents import UserProxyAgent

user_proxy = UserProxyAgent(
    name="user",
    code_execution_config={"executor": local_executor}
)
```

## Swarm 集群模式

### Swarm 模式核心理念

- **去中心化任务传递**：Swarm 团队模式摒弃了集中式调度，采用去中心化的任务传递方式。每个智能体通过 HandoffMessage 主动将任务传递给下一个合适的协作者，使任务流转更加灵活。
- **流程透明与可追溯**：与集中式调度相比，Swarm 模式通过显式的移交消息，使任务传递过程更加透明和可追溯。
- **对模型能力的要求**：Swarm 模式要求底层模型具备可靠的工具调用能力。

### 旅行客服场景

```
用户咨询 → 旅行客服（接收请求，判断类型）
              ├── 退款请求 → 退款处理员（调用退款工具，完成后反馈）
              └── 预订咨询 → 预订专员（处理预订，反馈结果）
```

退款处理员调用退款工具完成退款后，将结果反馈给旅行客服，客服总结并结束对话，形成完整的任务闭环。

```python
from autogen_agentchat.agents import handoff

# Agent A 移交任务给 Agent B
await agent_a.send(
    handoff(message="任务移交", target=agent_b)
)
```

## Magentic-One

### 概述

Magentic-One 是基于 AutoGen 框架构建的预构建通用任务解决专家系统，具备处理复杂任务的能力。既可以作为强大的工具直接使用，也是学习复杂多智能体设计的最佳实践蓝图。

### 系统组件

| Agent | 功能 |
|-------|------|
| Orchestrator | 任务规划和协调，是领导角色，负责任务理解、分解和监督 |
| WebSurfer | 网页浏览与交互，能够控制浏览器状态，执行网页导航和内容阅读 |
| FileSurfer | 本地文件系统操作，支持读取多种文件格式、浏览目录结构 |
| Coder | 代码编写和数据分析，根据任务需求生成 Python 脚本等代码 |
| ComputerTerminal | 为 Coder 提供安全的代码执行环境，确保代码安全运行 |

### 编排者双循环机制

- **内循环执行与反思**：编排者的内循环围绕进度分类账进行任务执行与即时反思，快速迭代以确保任务按计划推进
- **外循环规划调整**：当内循环陷入停滞时，编排者启动外循环，重新审视任务分类账，调整宏观计划以应对复杂情况
- **自适应能力**：双循环机制赋予 Magentic-One 强大的自适应能力

### 使用示例

```python
from autogen_ext.agents.magentic_one import MagenticOne
import logging

async def main():
    async with MagenticOne(log_level=logging.INFO) as agent:
        result = await agent.run(
            task="帮我分析这个 CSV 文件并生成报告"
        )
        print(result)

asyncio.run(main())
```

## GraphFlow 工作流编排

### GraphFlow 设计哲学

GraphFlow 作为确定性流程引擎，通过有向图精确控制智能体执行顺序，适用于企业级复杂流程。GraphFlow 与自由对话模式互为补充，满足不同场景需求。

### 核心组件与解耦思想

- **DiGraphBuilder**：负责构建工作流蓝图，添加智能体节点和定义执行路径
- **Node**：代表智能体
- **Edge**：定义节点间的执行依赖关系
- **执行图与消息图分离**：GraphFlow 实现执行图与消息图解耦，提升系统灵活性和可维护性

```python
from autogen_agentchat.graph import DiGraphBuilder, END

builder = DiGraphBuilder()

# 添加节点
builder.add_node(agent1, name="step1")
builder.add_node(agent2, name="step2")

# 添加边
builder.add_edge("step1", "step2")
builder.add_edge("step2", END)

graph = builder.build()
```

### 条件边与动态路由

通过 lambda 条件函数实现动态路由控制，将自然语言输出转化为布尔路由信号，实现流程的动态分支和跳转。结合 MaxMessageTermination 机制，从底层防止无限循环发生。

```python
builder.add_conditional_edge(
    "evaluate",
    lambda state: "approve" if state["score"] > 0.8 else "revise"
)
```

### 并行与合流

```python
# 并行处理
builder.add_edge("analyze", ["content_review", "style_review"])

# 合流（AND 逻辑）
builder.add_edge(["content_review", "style_review"], "finalize")
```

## 小结

AutoGen 提供了强大的多智能体协作能力。从双 Agent 对话到群聊，从 Swarm 到 Magentic-One，可以构建各种复杂度的多 Agent 应用。GraphFlow 则提供了确定性工作流的能力，与动态 Agent 协作形成互补。AgentChat 提供高层抽象方便快速开发，Core 提供底层控制实现极致定制。
