# 5.2 LangChain RAG 应用开发

## RAG 概述

### 什么是 RAG

**检索增强生成**（Retrieval-Augmented Generation，RAG）是一种结合信息检索与大语言模型生成的混合架构。它的核心思想是：当 LLM 需要回答问题时，先从外部知识源中检索相关信息，再将检索到的内容作为上下文提供给 LLM，由 LLM 基于这些真实、权威的知识生成回答。

RAG 的价值在于解决大语言模型的固有缺陷：知识过时、幻觉问题、以及对私有知识的无知。

### RAG 工作原理

RAG 的工作流程分为两个阶段：

**索引阶段**（离线）：将外部文档加载进来，对长文本进行分割，将每段文本通过嵌入模型转换为向量，存储到向量数据库中，并建立索引。

**检索与生成阶段**（在线）：用户提问时，先将问题本身也转换为向量，然后在向量数据库中进行相似度搜索，召回与问题最相关的 Top-K 段文本，最后将检索结果与问题一起构成提示词，交给 LLM 生成最终回答。

### RAG 解决了什么问题

传统大语言模型存在三大痛点：知识截止日期限制（无法获取最新信息）、幻觉倾向（生成看似合理但错误的内容）、缺乏私有知识（无法回答企业内部专有数据）。RAG 通过外挂知识库的方式，为 LLM 提供了一个"实时外脑"，在生成答案之前先检索真实资料，从根本上解决了这些问题。

### RAG 的应用场景

RAG 广泛应用于企业知识管理（如内部文档问答）、智能客服（如产品使用问题解答）、法律咨询（如法律条文检索）、医疗问答（如病历摘要与知识查询）等对准确性和时效性要求高的专业领域。

## LangChain 核心价值

### 为什么选择 LangChain 开发 RAG

从零开发一个完整的 RAG 应用，需要解决五大问题：繁杂的数据接入（各种格式的文档如何加载）、复杂的文本处理（如何将长文档切成合适的块）、多样的模型与接口（如何调用嵌入模型和 LLM）、专业的向量存储（如何选择和操作向量数据库）、以及如何将所有环节串联起来形成完整的数据流。LangChain 正是为解决这些问题而生的框架，它将 RAG 开发中的每一个环节都抽象为标准化组件，开发者只需要关注数据流本身。

### LangChain 模块化架构

LangChain 的 RAG 开发依赖两大核心模块：`langchain-core` 提供了所有组件的基础接口和抽象，`langchain-community` 则集成了数百种第三方服务（包括文档加载器、向量数据库、嵌入模型等）的实现。这种分层设计保证了接口的稳定性，同时借助社区力量不断扩展生态。

LangChain 在 RAG 开发中的优势体现在四个方面：流式处理支持（用户可以实时看到生成过程）、异步与并行执行（提升大规模数据处理效率）、可观察性与调试（方便追踪每个环节的输入输出）、以及标准化的组件接口（在不同技术方案之间切换时无需改动业务代码）。

## LangChain RAG 核心流程与组件

### 表达式语言（LCEL）

LangChain 表达式语言（LangChain Expression Language，LCEL）是 LangChain 的核心编程范式。它采用声明式的管道操作符 `|` 将多个组件串联起来，数据像流水一样从左向右流经每一个处理环节。

LCEL 的核心优势包括：原生支持流式处理（可以从第一个组件开始流式输出）、完整的异步支持（高性能生产环境必备）、可观察性（每个步骤都可以被追踪）、以及高度的可组合性（复杂流程可以用简单的方式拼接）。

一个最简单的 RAG 链形如：`prompt | model | output_parser`，分别代表提示词模板、模型、输出解析器。

### 文档加载器

文档加载器是 RAG 流程的入口，负责将各种格式的外部数据源加载为标准化的 Document 对象。每个 Document 对象包含两个核心属性：`page_content`（文档内容文本）和 `metadata`（元数据，如文档来源、页码、作者等）。

LangChain 的文档加载器遵循基础加载器接口规范，提供两种加载方式：贪婪加载一次性将所有文档读入内存（适合小规模数据），懒加载则按需逐个加载文档对象（适合大规模数据源）。

### 常用文档加载器对比

| 加载器 | 适用场景 | 特点 |
|--------|---------|------|
| TextLoader | 本地纯文本文件 | 简单直接，配置编码格式 |
| WebBaseLoader | 网页内容 | 依赖 urllib 和 BeautifulSoup，可提取元数据 |
| CSVLoader | CSV 表格文件 | 每行转换为一个 Document |
| PyPDFLoader | PDF 文档 | 支持按页或整体加载 |
| DirectoryLoader | 整个目录 | 支持 glob 模式匹配，可指定子加载器 |

```python
from langchain_community.document_loaders import TextLoader, WebBaseLoader, CSVLoader, PyPDFLoader, DirectoryLoader

# 文本加载器
loader = TextLoader("example.txt", encoding="utf-8")
docs = loader.load()

# 网页加载器
loader = WebBaseLoader("https://example.com/article")
docs = loader.load()

# CSV 加载器
loader = CSVLoader("data.csv")
docs = loader.load()

# PDF 加载器
loader = PyPDFLoader("document.pdf")
docs = loader.load()

# 目录加载器
loader = DirectoryLoader(
    "./documents",
    glob="**/*.txt",
    loader_cls=TextLoader
)
docs = loader.load()
```

## 文档分割

### 文本分割的重要性

文本分割（分块）是 RAG 流程中连接原始数据与向量知识库的关键环节。如果分割策略不当，后续的检索效率和生成质量都会受到严重影响。

文本分割的必要性体现在三个层面。首先，大语言模型的上下文窗口存在限制，超出部分的信息会被直接忽略，合理分块可以确保关键信息完整保留在窗口内。其次，小而精的文本块能够显著提升检索的准确率和召回率，因为块越小，语义越聚焦，与查询的相关性判断越精准。最后，分块策略直接影响嵌入成本、查询延迟和 LLM 的幻觉风险——块过大可能引入过多无关噪声，块过小则丢失上下文。

### 核心参数

文本分割有两个核心参数：**块大小**（chunk_size）定义了每个文本块的最大尺寸，通常以字符数或 Token 数为单位；**块重叠**（chunk_overlap）定义了相邻块之间重复内容的数量，保证关键信息不会因分割边界而被切断。

分块策略需要在三个目标之间权衡：检索精度（更小、更具主题特异性的块效果更好）、上下文丰富程度（更大的块包含更多上下文信息）、以及系统效率（大块数量少，查询更快）。

### 字符文本分割器

字符文本分割器是最基础的分块工具，它按照指定字符（如换行符 `\n\n`）作为分隔符进行分割，当分割后的块仍然超过块大小时，再按更小的单位继续分割。

```python
from langchain.text_splitter import CharacterTextSplitter

text_splitter = CharacterTextSplitter(
    separator="\n\n",
    chunk_size=1000,
    chunk_overlap=200
)
docs = text_splitter.split_documents(raw_docs)
```

这种分割器的优点是简单快速，局限性在于过于机械化，无法处理复杂文本结构，可能导致语义不完整的块。

### 递归字符文本分割器

递归字符文本分割器采用"分而治之"的递归策略，使用有序的分隔符列表（从大到小：段落 → 换行 → 句子 → 单词 → 字符）层层递进地分割文本，生成语义更连贯的文本块。

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    separators=["\n\n", "\n", "。", "！", "？", " ", ""],
    chunk_size=1000,
    chunk_overlap=200
)
docs = text_splitter.split_documents(raw_docs)
```

递归分割的逻辑是：首先尝试按最大分隔符（段落）分割，如果产生的块仍超过限制，就用第二大的分隔符继续分割，如此递归直到块大小合适为止。

### 按 Token 分割

Token 是大语言模型处理文本的基本单位，与字符数没有固定的换算比例。由于 LLM 的上下文窗口限制、训练数据量和 API 计费都以 Token 为单位计算，按 Token 分割能更精确地控制输入内容的大小。

```python
from langchain.text_splitter import TokenTextSplitter

text_splitter = TokenTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
docs = text_splitter.split_documents(raw_docs)
```

递归 Token 分割器融合了递归字符分割的逻辑和 Token 计数的精确性，既尊重文本结构，又确保块大小精确。

### 结构感知分割器

对于代码、Markdown、HTML 等具有结构化格式的文档，LangChain 提供了专用的结构感知分割器，能够保留原始结构信息。

```python
from langchain.text_splitter import (
    Language,
    RecursiveCharacterTextSplitter
)

# 代码分割器 - 以 Python 为例
python_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=500,
    chunk_overlap=50
)
code_chunks = python_splitter.split_documents(code_docs)

# Markdown 分割器
markdown_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.MARKDOWN,
    chunk_size=500,
    chunk_overlap=50
)
md_chunks = markdown_splitter.split_documents(md_docs)

# HTML 分割器
html_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.HTML,
    chunk_size=500,
    chunk_overlap=50
)
html_chunks = html_splitter.split_documents(html_docs)
```

代码分割器按照编程语言的分隔符规则进行分割，保留完整的函数和类定义；Markdown 分割器依据标题级别进行分割，并将标题层级附加到元数据中；HTML 分割器解析 DOM 树结构，按指定的标题标签分割。

## 文本嵌入

### 什么是向量

向量是具有大小和方向的几何对象。在计算机中，向量用有序数字列表表示，例如二维向量 (X, Y)。在机器学习中，向量通常有很高维度（比如 768 维、1024 维甚至更高），每个分量代表数据在特定维度上的特征值。

在向量空间中，距离近或方向相似的向量被认为语义相关——这是语义检索的数学基础。例如，"猫在沙发上睡觉"和"一只猫咪卧在沙发上休息"虽然文字不同，但语义相近，它们的向量在空间中的位置也会非常接近。

### 什么是文本嵌入

文本嵌入（Embedding）是将文本映射到高维向量空间的技术。语义相近的文本在该空间中位置靠近，语义相反的文本则距离较远。文本嵌入催生了语义检索能力，使系统能理解用户意图，找到与查询语义相关的内容，而不仅是关键词字面匹配。

### LangChain 统一嵌入接口

LangChain 为各种嵌入模型提供了统一的接口，核心方法有两个：`embed_documents` 用于批量将多个文档嵌入为向量，`embed_query` 用于将用户查询嵌入为向量。这种非对称搜索的设计（查询和文档使用同一模型但不同方法）确保了检索的准确性。

```python
from langchain_community.embeddings import DashScopeEmbeddings

# 使用阿里云百炼平台的嵌入模型
embeddings = DashScopeEmbeddings(
    model="text-embedding-v4",
    dashscope_api_key="your-api-key"
)

# 嵌入文档（批量）
document_vectors = embeddings.embed_documents([
    "文档内容1",
    "文档内容2"
])

# 嵌入查询（单个）
query_vector = embeddings.embed_query("用户的问题是什么")
```

### 本地嵌入模型

在数据隐私要求高或需要控制成本时，可以选择使用本地部署的嵌入模型。

```python
from langchain_huggingface import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="./Qwen3-Embedding-0.6B",
    model_kwargs={"device": "cuda"},  # 使用 GPU
    encode_kwargs={"normalize_embeddings": True}  # 归一化处理
)
```

本地嵌入的优势体现在四个方面：数据不出本地（隐私安全）、离线可用（不依赖外部网络）、成本可控（一次性硬件投入）、以及可针对私有数据微调。

### GPU 加速嵌入

对于大规模文档处理，使用 GPU 加速可以显著提升嵌入效率。原理是：将数据从主机内存复制到 GPU 显存，利用 GPU 的并行计算能力同时对批次中的多个文本进行嵌入计算，最后将结果返回主机内存。

```python
import torch

# 验证 GPU 是否可用
print(f"CUDA available: {torch.cuda.is_available()}")

# 在代码中指定使用 GPU
embeddings = HuggingFaceEmbeddings(
    model_name="./Qwen3-Embedding-0.6B",
    model_kwargs={"device": "cuda"},
    encode_kwargs={"normalize_embeddings": True}
)
```

## 向量存储

### 向量存储的重要性

向量存储是 RAG 系统的"记忆中枢"，负责存储海量知识库信息，并高效检索与问题意图契合的内容片段。向量存储的检索效率和质量直接影响 RAG 系统输出内容的准确性、深度和可靠性。

传统关系型数据库依赖精确的关键词或标签匹配，无法理解深层语义。向量数据库通过深度学习模型将文本转换为向量，将"字符串匹配"提升到"语义概念匹配"。

### 向量搜索原理

向量搜索的本质是"相似性搜索"：在庞大向量集合中快速找出与查询向量最相似的 N 个向量，通过计算向量之间的距离来衡量内容相似度。

嵌入模型的质量是向量搜索效果的天花板。优秀的嵌入模型能生成结构优良、语义区分度高的向量空间，使语义相近的内容向量紧密相连。

### 核心度量方法

| 度量方法 | 公式 | 特点 | 适用场景 |
|---------|------|------|---------|
| 欧氏距离 | 两点间直线距离 | 关注绝对位置差异，受向量长度影响 | 某些特定场景 |
| 余弦相似度 | 向量夹角余弦值 | 只关心方向，不受长度影响，范围 [-1, 1] | 文本语义检索（推荐） |

在实际处理文本数据时，**余弦相似度**通常更受青睐，因为它能更纯粹地衡量语义相关性，而忽略篇幅长短等外部因素。

### 近似最近邻搜索（ANN）

精确最近邻搜索需要将查询向量与数据库中所有向量逐一比较，在海量数据场景下效率极低。近似最近邻搜索（ANN）通过预先构建索引，将相似向量组织成"社区"，查询时直接定位到最可能包含答案的区域，以微小精度损失换取数量级上的速度提升。

### LangChain 与 FAISS 实战

FAISS（Facebook AI Similarity Search）是由 Meta 开源的高性能相似性搜索库，使用 C++ 实现提供 Python 接口，在LangChain 中使用需要配合嵌入模型一起实例化。

```python
from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS

# 1. 加载文档
loader = TextLoader("tutorial.txt", encoding="utf-8")
docs = loader.load()

# 2. 分割文档
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

# 3. 创建嵌入模型
embeddings = HuggingFaceEmbeddings(
    model_name="./Qwen3-Embedding-0.6B",
    model_kwargs={"device": "cuda"},
    encode_kwargs={"normalize_embeddings": True}
)

# 4. 创建 FAISS 向量存储
vectorstore = FAISS.from_documents(documents=chunks, embedding=embeddings)

# 5. 保存到本地
vectorstore.save_local("./faiss_index")

# 6. 从本地加载
loaded_vectorstore = FAISS.load_local(
    "./faiss_index",
    embeddings,
    allow_dangerous_deserialization=True
)

# 7. 执行相似性搜索
results = loaded_vectorstore.similarity_search("查询内容", k=3)
```

### LangChain 与 ChromaDB 实战

ChromaDB 是一个为 AI 应用而生的开源向量数据库，提供简单直观的 Python API，内置数据持久化、元数据过滤等功能。

```python
from langchain_chroma import Chroma
from langchain_huggingface import HuggingFaceEmbeddings

# 创建并持久化
chroma_db = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)

# 从磁盘加载
loaded_chroma = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings
)

# 执行搜索

```

ChromaDB 内置自动持久化功能，在创建时会自动保存数据到指定目录，无需手动调用 save 方法。

## 检索

### 检索器接口

检索器（Retriever）是 LangChain 中负责从知识库中获取相关信息的关键组件。它接受用户查询，返回与查询相关的文档列表。LangChain 使用高度抽象的接口设计，屏蔽了底层向量数据库的差异，上层应用只需调用统一的检索接口。

```python
# 将向量存储转换为检索器
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 5}  # 返回最相似的 5 个文档
)

# 执行检索
results = retriever.get_relevant_documents("用户的问题")
```

### 检索的搜索类型

**相似性搜索**是最基础的检索方式，将查询转换为向量，计算与知识库中文档向量的相似度，返回最接近的 k 个文档。这种方式简单高效，适用于大多数通用问答场景。

**最大边际相关性（MMR）** 算法在提升检索结果多样性方面表现优异。传统相似性搜索返回的文档往往内容高度重复，MMR 在选择相关文档时不仅考虑与查询的相关性，还考虑与已选文档的差异性，从而在相关性和多样性之间取得平衡。

```python
# 相似性搜索
results = vectorstore.similarity_search("查询", k=3)

# MMR 搜索
results = vectorstore.max_marginal_relevance_search(
    "查询",
    k=3,              # 最终返回 3 个
    fetch_k=10,       # 先取 10 个候选
    lambda_mult=0.5   # 0=最 diversity，1=最相关
)

# 带相似度阈值的搜索
results = vectorstore.similarity_search_with_score(
    "查询",
    k=5,
    score_threshold=0.5  # 只返回相似度 > 0.5 的结果
)
```

## 构建完整 RAG 链

### 使用 LCEL 构建 RAG 链

LangChain 的 LCEL 让构建完整 RAG 流程变得极为简洁：

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

# 构造提示词模板
prompt = ChatPromptTemplate.from_template(
    """基于以下参考资料回答用户的问题。如果资料中没有相关信息，请如实告知。

参考资料：
{context}

问题：{question}"""
)

# 串联为链：检索 -> 组合提示词 -> LLM 生成
rag_chain = (
    {
        "context": retriever,      # 自动检索相关文档
        "question": RunnablePassthrough()
    }
    | prompt                       # 构建提示词
    | llm                          # LLM 生成回答
    | StrOutputParser()            # 解析输出为字符串
)

# 执行
result = rag_chain.invoke("今天北京的天气怎么样？")
```

### LCEL 的优势

LCEL 采用声明式管道操作符，数据流清晰可见。上例中，`retriever` 接收问题并检索相关文档，`prompt` 接收 `context` 和 `question` 构造完整提示词，`llm` 生成回答，最后 `StrOutputParser` 解析输出。每个环节都可以独立测试、替换或添加新功能。

### 并行与条件分支

LCEL 支持并行处理和条件分支，可以应对复杂业务场景：

```python
from langchain_core.runnables import RunnablePassthrough

# 并行检索多个知识库
combined_rag = (
    {
        "context1": retriever1,
        "context2": retriever2,
        "question": RunnablePassthrough()
    }
    | prompt
    | llm
    | StrOutputParser()
)
```

## 对话式 RAG

### 无状态 RAG 的局限

简单的 RAG 只能处理单轮问答。当用户追问"那第二点呢？"时，RAG 系统无法理解"那第二点"指向的是上一轮回答中的第二个要点。这是在真实对话场景中必须解决的问题。

### 引入对话历史

解决无状态局限的核心思路是：当用户提出新问题时，将之前的历史对话一并考虑进来。具体做法有两种：方案一是将完整历史对话拼接到查询中一起做嵌入检索；方案二是训练一个专门的"查询重写"模型，将历史对话与当前问题改写为一个独立的查询。

第一种方案简单但有明显缺陷：对话历史累积会导致成本和延迟急剧上升，同时大量历史信息会成为干扰噪声，降低检索效率。因此，更推荐使用查询重写的方式。

### LangGraph 构建对话式 RAG

LangGraph 是 LangChain 生态中专门用于构建复杂 Agent 流程的框架，擅长处理循环、条件分支、并行执行和持久化状态。它用状态图的方式描述 RAG 流程：节点（Node）代表工作单元，边（Edge）代表节点间的控制流，状态（State）是在各节点间流转的核心数据对象。

```python
from typing import List, TypedDict, Annotated
from langchain_core.documents import Document
from langchain_core.messages import BaseMessage, AIMessage, HumanMessage
import operator

# 定义图状态
class GraphState(TypedDict):
    messages: Annotated[List[BaseMessage], operator.add]  # 对话历史（累积）
    question: str       # 用户原始问题
    standalone_question: str  # 重写后的独立问题
    documents: List[Document]  # 检索到的文档
    generation: str     # 最终回答

# 查询重写节点
def rewrite_node(state: GraphState) -> dict:
    """根据对话历史，将用户问题改写为独立查询"""
    messages = state["messages"]
    question = state["question"]

    if len(messages) <= 1:
        # 首轮对话无需重写
        return {"standalone_question": question}

    rewrite_prompt = ChatPromptTemplate.from_messages([
        ("system", "根据对话历史，将最新问题改写为独立问题，使其无需上下文也能理解。"),
        MessagesPlaceholder(variable_name="chat_history"),
        ("human", "{question}"),
    ])
    rewriter_chain = rewrite_prompt | llm | StrOutputParser()
    standalone = rewriter_chain.invoke({
        "chat_history": messages[:-1],
        "question": question
    })
    return {"standalone_question": standalone}

# 检索节点
def retrieve_node(state: GraphState) -> dict:
    """从向量数据库中检索相关文档"""
    docs = compression_retriever.invoke(state["standalone_question"])
    return {"documents": docs}

# 生成节点
def generate_node(state: GraphState) -> dict:
    """根据检索结果生成最终回答"""
    docs = state.get("documents", [])
    question = state["standalone_question"]

    if not docs:
        # 无相关文档，转为闲聊
        prompt = ChatPromptTemplate.from_messages([
            ("system", "你是一个乐于助人的AI助手。"),
            MessagesPlaceholder(variable_name="chat_history"),
        ])
        chain = prompt | llm | StrOutputParser()
        response = ""
        for chunk in chain.stream({"chat_history": state["messages"]}):
            response += chunk
    else:
        context = "\n\n".join([doc.page_content for doc in docs])
        prompt = ChatPromptTemplate.from_template(
            "根据以下资料回答问题。如果资料中没有提到，请说不知道。\n\n资料：{context}\n\n问题：{question}"
        )
        chain = prompt | llm | StrOutputParser()
        response = ""
        for chunk in chain.stream({"context": context, "question": question}):
            response += chunk

    # 将回答加入历史
    new_messages = state["messages"] + [AIMessage(content=response)]
    return {"generation": response, "messages": new_messages}

# 条件路由：判断是否需要检索
def agent_router(state: GraphState) -> str:
    decision = (llm | StrOutputParser()).invoke(
        f"判断以下问题是否需要从知识库检索回答，只能回答 'retrieve' 或 'chat'：{state['standalone_question']}"
    )
    return "retrieve" if "retrieve" in decision.lower() else "generate"

# 构建图
from langgraph.graph import StateGraph, END
workflow = StateGraph(GraphState)
workflow.add_node("rewrite", rewrite_node)
workflow.add_node("retrieve", retrieve_node)
workflow.add_node("generate", generate_node)
workflow.set_entry_point("rewrite")

workflow.add_conditional_edges(
    "rewrite",
    agent_router,
    {"retrieve": "retrieve", "generate": "generate"}
)
workflow.add_edge("retrieve", "generate")
workflow.add_edge("generate", END)

app = workflow.compile()
```

上述代码构建了一个完整的对话式 RAG 应用：用户提问后，先经过 `rewrite` 节点将问题改写为独立查询（考虑历史对话），然后通过 `agent_router` 判断是否需要检索知识库。如果需要，则执行 `retrieve` 再 `generate`；如果判断为闲聊，则直接 `generate`。最终回答会累积到 `messages` 状态中，供下一轮对话使用。

## 高级检索策略

### 两阶段检索：召回与精排

在召回阶段，采用多查询检索器（MultiQueryRetriever）从不同角度生成多个查询变体，提升召回率。但这种方式可能会引入不相关的噪声数据。

在精排阶段，使用重排模型（Cross-Encoder Reranker）对召回结果进行二次排序。交叉编码器将查询和文档作为配对输入，通过神经网络判断它们之间的相关性，最终返回最相关的结果。

```python
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import HuggingFaceCrossEncoder

# 创建重排模型
reranker = CrossEncoderReranker(
    model=HuggingFaceCrossEncoder(model_name="BAAI/bge-reranker-base"),
    top_n=3  # 重排后保留前 3 个
)

# 构建压缩检索器
compression_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=base_retriever
)
```

这种两阶段策略先用向量检索快速从海量数据中召回候选文档，再用重型但更精准的交叉编码器进行精细化排序，在速度和精度之间取得平衡。
