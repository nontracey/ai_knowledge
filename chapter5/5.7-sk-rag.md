# 5.7 Semantic Kernel RAG

## SK RAG 架构与抽象层

### 抽象层设计理念

Semantic Kernel 为 RAG 应用提供了现代化、企业级的抽象层。相比 LangChain 的高度封装，SK 的设计更强调精细、解耦和类型安全。这种架构使得系统更加健壮、可维护，同时分离了数据存储、嵌入生成和信息检索的关注点，提升了模块化和可测试性。

### 数据模型映射

通过定义 C# 类和添加特性，可以将数据模型映射到向量数据库的模式。例如，使用 `VectorStoreKey`、`VectorStoreData`、`VectorStoreVector` 等特性，能够简化与数据库的交互，避免编写复杂的数据库迁移代码，提高代码的可维护性和可扩展性。

```csharp
public class DocumentChunk
{
    [VectorStoreKey]
    public string Id { get; set; }

    [VectorStoreData]
    public string Content { get; set; }

    [VectorStoreVector(1536)]
    public ReadOnlyMemory<float> Embedding { get; set; }

    [VectorStoreData]
    public string Source { get; set; }

    public DateTime CreatedAt { get; set; }
}
```

这种模型优先的设计理念，让开发者能够专注于业务逻辑，而不是底层的数据库操作。

### 技术选型灵活性

SK 的架构设计增强了技术选型的灵活性，允许开发者根据项目需求选择合适的数据库和嵌入模型，同时保持系统的稳定性和可扩展性。通过更换向量数据库连接器，可以无缝迁移到底层不同的数据库，无需改动上层业务代码。

## RAG 完整生命周期

### 生命周期阶段

RAG 应用的核心生命周期包括六个阶段：

1. **数据加载与预处理**：读取外部文档，解析不同格式（PDF、Word、HTML 等）
2. **文本分块**：将长文档分割成适合嵌入和检索的小块
3. **嵌入**：使用嵌入模型将文本块转换为高维向量
4. **索引**：将向量和元数据存储到向量数据库
5. **检索**：根据用户查询从向量数据库中召回相关文档
6. **增强生成**：将检索结果作为上下文交给 LLM 生成回答

每个阶段都有明确的作用和输入输出定义。

### 与 LangChain 的生命周期对比

| 阶段 | Semantic Kernel | LangChain |
|------|-----------------|-----------|
| 数据加载 | 手动编码（如 File.ReadAllTextAsync） | 庞大的 DocumentLoader 生态 |
| 文本分块 | TextChunker 静态类 | 多种 TextSplitter 实现 |
| 向量存储 | 抽象的强类型，与 C# 数据模型绑定 | 统一的向量存储包装器接口 |
| 检索生成 | SearchAsync → 手动构建上下文 → InvokePromptAsync | 一行代码构建完整 RAG 链 |

## 构建标准 RAG 应用流程

### 环境准备

```csharp
// 创建内核
var builder = Kernel.CreateBuilder();
builder.AddOpenAIChatCompletion("gpt-4", apiKey);
builder.AddOpenAITextEmbeddingGeneration("text-embedding-3-small", apiKey);
var kernel = builder.Build();
```

### 文本预处理与分块

文本分块是 RAG 流程中的关键步骤，能够将长文本分割成适合嵌入和检索的小块，从而提高检索的准确性和效率。

```csharp
using Microsoft.SemanticKernel.Text;

// 使用 TextChunker 进行分块
var chunks = TextChunker.SplitPlainTextParagraphs(
    text: documentContent,
    maxTokensPerParagraph: 1000,
    overlapTokens: 200
);

// 按行分割并重组段落
var lineChunks = TextChunker.SplitTextLines(
    text: documentContent,
    maxTokensPerLine: 100
);
```

### 自定义 Token 计算器

默认的 Token 计算器在处理中文分词时存在局限性，通过自定义 Token 计算器可以解决这一问题：

```csharp
public class ChineseTokenCalculator : ITokenCounter
{
    public int CountTokens(string text)
    {
        // 简单估算：中文每个字符约 1.5 个 Token
        return (int)(text.Length * 1.5);
    }
}
```

### 封装不稳定依赖

为了隔离变更风险，可以创建稳定的接口和包装类：

```csharp
public interface ITextSplitter
{
    IEnumerable<string> Split(string text);
}

public class SemanticKernelTextSplitter : ITextSplitter
{
    public IEnumerable<string> Split(string text)
    {
        return TextChunker.SplitPlainTextParagraphs(
            text,
            maxTokensPerParagraph: 1000,
            overlapTokens: 200
        );
    }
}
```

## 向量存储与模型映射

### 向量存储设计

Semantic Kernel 的向量存储设计通过定义 C# 类和添加特性，声明性地描述数据模型如何映射到向量数据库的模式：

```csharp
public class Article
{
    [VectorStoreKey]
    public string Id { get; set; }

    [VectorStoreData]
    public string Title { get; set; }

    [VectorStoreData]
    public string Content { get; set; }

    [VectorStoreVector(1536)]
    public ReadOnlyMemory<float> ContentEmbedding { get; set; }

    [VectorStoreData]
    public string Author { get; set; }

    [VectorStoreData]
    public DateTime PublishDate { get; set; }
}
```

### 核心特性解读

- `[VectorStoreKey]`：标记为主键，唯一标识每条记录
- `[VectorStoreData]`：标记为数据字段，会被存储到数据库
- `[VectorStoreVector(dim)]`：标记为向量字段，dim 指定向量维度

### 使用 ChromaDB 存储

```csharp
using Microsoft.SemanticKernel.Connectors.Chroma;

// 创建 ChromaDB 向量存储
var vectorStore = new ChromaVectorStore(
    embeddingGenerator,
    new ChromaClient("http://localhost:8000"),
    "articles"
);

await vectorStore.CreateCollectionAsync();

// 添加文档
var article = new Article
{
    Id = Guid.NewGuid().ToString(),
    Title = "RAG 技术详解",
    Content = "检索增强生成是一种混合架构...",
    ContentEmbedding = await embeddingGenerator.GenerateEmbeddingAsync("RAG 技术详解")
};

await vectorStore.UpsertAsync(article);
```

## 文本搜索服务

### ITextSearch 接口

Semantic Kernel 提供了 `ITextSearch` 接口，将复杂的向量检索过程封装成一个简单的基于自然语言的搜索操作：

```csharp
var textSearch = kernel.GetService<ITextSearch>();

var results = await textSearch.SearchAsync(
    "RAG 技术的优势",
    options: new TextSearchOptions
    {
        Top = 3,
        Skip = 0
    }
);

foreach (var result in results.Results)
{
    Console.WriteLine($"标题: {result.Text}");
    Console.WriteLine($"相似度: {result.Score}");
}
```

### TextSearchResultValue 特性

通过在数据模型上添加 `TextSearchResultValue` 等特性，可以将检索到的自定义数据模型转换为标准格式：

```csharp
public class SearchableArticle
{
    [TextSearchResultValue]
    public string Text { get; set; }  // 作为搜索结果展示的文本

    [VectorStoreData]
    public string Title { get; set; }

    [VectorStoreData]
    public string Content { get; set; }
}
```

### 分层抽象的优势

这种分层抽象的设计实现了上层应用代码与底层数据库技术的解耦。通过更换向量数据库连接器，可以无缝迁移到底层不同的数据库（如从 ChromaDB 切换到 Qdrant 或 Milvus），无需改动业务逻辑代码。

## SK RAG 与 LangChain RAG 对比

### 特性对比表

| 特性/组件 | Semantic Kernel | LangChain | 对比分析 |
|-----------|-----------------|-----------|---------|
| 数据加载 | 无内置加载器生态，需手动编码 | 庞大的 DocumentLoader 生态，支持数百种数据源 | LangChain 在数据接入的便捷性和广度上具有明显优势 |
| 文本分块 | 提供 TextChunker 静态类，支持按行、段落和重叠分块 | 丰富的 TextSplitter 抽象和实现 | LangChain 的分块策略更多样化，适应性更强 |
| 向量存储 | 采用抽象的强类型，与 C# 数据模型紧密绑定 | 提供统一的向量存储包装器接口 | SK 提供了编译时类型安全和更好的 IDE 支持 |
| RAG 流程构建 | 手动流程：SearchAsync → 手动构建上下文 → InvokePromptAsync | 高度封装：提供高级抽象，一行代码即可构建完整 RAG 链 | LangChain 在 RAG 模式的封装程度上远高于 SK |

### 设计理念差异

- **Semantic Kernel** 的设计理念是编排，强调强类型、面向对象、依赖注入和关注点分离，适合构建长期维护、稳定性和可扩展性要求高的生产级 AI 应用。

- **LangChain** 的设计理念更侧重于灵活性和快速实验，采用函数式、声明式的编程范式，适合快速原型验证和频繁调整的场景。

### 开发体验对比

| 维度 | Semantic Kernel | LangChain |
|------|-----------------|-----------|
| 类型安全 | 编译时检查，IDE 支持好 | 动态类型，运行时才能发现错误 |
| 可观测性 | 支持 OpenTelemetry | LangSmith 平台支持 |
| 生态系统 | .NET 生态，AI 社区相对较小 | Python 生态，AI 社区庞大 |
| 学习曲线 | 对 .NET 开发者友好 | 对 Python 开发者友好 |

### 场景选择建议

**选择 Semantic Kernel 的场景：**
- 深度集成 .NET 企业应用
- 追求稳定性和可维护性
- 构建标准化的 AI 服务
- 团队熟悉 C# 和 .NET 技术栈

**选择 LangChain 的场景：**
- 快速原型和研究探索
- 需要利用丰富的生态系统
- Python 技术栈主导
- 频繁调整和实验

## SK Agent 框架构建智能 RAG

### Agentic RAG 核心思想

**传统 RAG 的局限性**：传统 RAG 缺乏动态决策能力，无法处理模糊查询和多步推理任务。它总是按照"检索 → 生成"的固定流程执行，即使面对不需要检索的简单问题也是如此。

**Agentic RAG 的引入**：Agentic RAG 引入智能体（Agent）的概念，通过推理与规划、工具选择与使用、迭代执行来处理更复杂的任务。这种动态的、基于推理的循环提升了 RAG 系统的灵活性和智能性。

### 标准 RAG vs 智能 RAG

| 类型 | 模式 | 工作方式 | 适用场景 |
|------|------|---------|---------|
| 标准 RAG | 推模式 | 总是先执行检索，将结果推送给语言模型 | 简单的事实问答 |
| 智能 RAG | 拉模式 | Agent 根据内部的推理，自主决定是否需要从知识库中拉取信息 | 复杂推理、多步规划 |

### TextSearchProvider 实现原理

TextSearchProvider 实现了 `IAIContextProvider` 接口，在 Agent 调用语言模型之前被触发，分析用户输入并进行相关性搜索：

```csharp
public class KnowledgeBaseSearchProvider : IAIContextProvider
{
    private readonly ITextSearch _textSearch;

    public KnowledgeBaseSearchProvider(ITextSearch textSearch)
    {
        _textSearch = textSearch;
    }

    public async Task<IEnumerable<ChunkedText>> GetRelevantTextAsync(
        string userInput,
        KernelArguments arguments,
        CancellationToken cancellationToken = default)
    {
        // 执行向量搜索
        var searchResults = await _textSearch.SearchAsync(
            userInput,
            new TextSearchOptions { Top = 5 }
        );

        // 转换为 ChunkedText 格式
        return searchResults.Results.Select(r => new ChunkedText(r.Text));
    }
}
```

### Agentic RAG 的优势

1. **动态决策**：Agent 可以根据问题的复杂度决定是否需要检索
2. **迭代优化**：如果首次检索结果不理想，Agent 可以重新检索
3. **多步推理**：Agent 可以将复杂问题分解为多个步骤，每个步骤可能涉及不同的检索策略

### 完整示例

```csharp
// 创建内核并配置服务
var builder = Kernel.CreateBuilder();
builder.AddOpenAIChatCompletion("gpt-4", apiKey);
builder.AddOpenAITextEmbeddingGeneration("text-embedding-3-small", apiKey);
var kernel = builder.Build();

// 注册文本搜索服务
kernel.Services.AddSingleton<ITextSearch>(sp =>
    new TextSearchService(vectorStore, embeddingGenerator));

// 创建 Agent
var agent = new ChatCompletionAgent
{
    Kernel = kernel,
    Instructions = @"你是一个智能助手。对于用户的问题，你需要判断：
1. 如果问题涉及具体事实或知识，先从知识库检索相关信息
2. 如果是闲聊或开放性问题，直接回答
3. 回答时，优先使用检索到的知识，结合你的理解给出完整答案。"
};

// 与 Agent 对话
var agentThread = new ChatHistoryAgentThread();
var response = await agent.InvokeAsync(
    new ChatMessageContent(AuthorRole.User, "RAG 和 Agent 有什么关系？"),
    agentThread
);
```

## 小结

Semantic Kernel 提供了企业级的 RAG 实现方式，强类型和面向对象的特性使其在 .NET 环境中具有独特优势。相比 LangChain 的高度封装，SK 提供了更底层的控制，让开发者能精细地管理流程的每一步。

SK RAG 的核心价值体现在：强类型带来的编译时安全保障、依赖注入带来的可测试性、以及与 .NET 企业生态的深度集成。Agentic RAG 则通过赋予 Agent 自主决策能力，处理更复杂的问答场景，实现"按需检索"而非"强制检索"。在实际项目中，可以根据团队技术栈和项目需求选择合适的框架，也可以将两者结合使用，兼顾灵活性与稳定性。
